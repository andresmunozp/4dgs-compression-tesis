# 3.4 Rendering and Reconstruction

El proceso de renderizado y reconstrucción de modelos 4DGS descomprimidos es fundamental para convertir los datos comprimidos en representaciones visuales interactivas. Este proceso se divide en tres fases: descompresión del archivo, reconstrucción de parámetros gaussianos y carga en motores de visualización.

---

## 3.4.1 Proceso de Descompresión

El proceso de descompresión implementado en `decompress.py` se estructura en **tres fases claramente diferenciadas**, cada una con responsabilidades específicas:

### Fase 1: Reagrupamiento de Chunks

**Objetivo**: Reconstruir el archivo completo `.4dgs` a partir de los chunks transmitidos.

**Proceso**:
```
Chunks recibidos (chunk_00000_of_00042.4dgsc)
    ↓
ModelAssembler (validación de integridad)
    ↓
Verificación CRC32 por chunk
    ↓
Reordenamiento secuencial
    ↓
Archivo completo .4dgs
```

**Detalles técnicos**:
1. Lee todos los chunks desde el directorio especificado
2. Valida la integridad de cada chunk mediante CRC32:
   ```
   header + payload → crc32(header + payload) vs stored crc32
   ```
3. Verifica que la secuencia sea completa (chunk_id ∈ [0, total-1])
4. Concatena los payloads en orden para reconstruir el archivo original
5. **Latencia típica**: 100-500 ms (según número y tamaño de chunks)

**Ventajas de esta separación**:
- Permite reenvío selectivo de chunks corruptos sin retransmisión completa
- Validación temprana de errores de transmisión
- Métricas de confiabilidad de red (tasa de chunks perdidos/corruptos)

---

### Fase 2: Decodificación (Descompresión)

**Objetivo**: Aplicar las estrategias de compresión en orden **inverso** para restaurar los datos gaussianos originales.

**Implicaciones de Compresión Lossy vs. Lossless**:

La descompresión debe distinguir entre estrategias **reversibles** e **irreversibles** (ver sección 3.3.2):

- **Estrategias lossless** (Entropy Coding): Son **completamente reversibles**. Los datos descomprimidos son idénticos bit a bit a los originales antes de la compresión entrópica. El algoritmo de decodificación (zlib.decompress, zstd.decompress) garantiza reconstrucción exacta sin pérdida de información.

- **Estrategias lossy** (Pruning, SH Reduction, Quantization, HexPlane SVD): Son **irreversibles por naturaleza**. La información descartada durante la compresión (gaussianas eliminadas, coeficientes SH truncados, precisión numérica reducida) **no puede recuperarse** durante la descompresión. El proceso de "descompresión" para estas estrategias consiste en:
  - **Dequantización**: Aproxima valores originales aplicando factores de escala (float16→float32 introduce error ~0.0005)
  - **SH Expansion**: Rellena coeficientes faltantes con ceros (sin recuperar información original)
  - **Pruning Reconstruction**: Las gaussianas eliminadas permanecen ausentes (pérdida permanente)
  - **HexPlane SVD**: Reconstruye W ≈ U × Σ_k × V^T con error controlado por rank k

Esta **asimetría fundamental** significa que model_descomprimido ≠ model_original para perfiles Balanced y Aggressive, pero el error introducido está **controlado y medido** mediante métricas perceptuales (PSNR, SSIM, LPIPS) para garantizar que la degradación visual sea mínima.

**Orden de aplicación (inverso)**:
```
Archivo .4dgs (comprimido)
    ↓
[1] Entropy Decoding (zlib/zstd/lz4)
    ↓ (recupera datos cuantizados/podados)
[2] HexPlane Decompression (SVD/quantize reversal)
    ↓ (restaura pesos de red de deformación)
[3] Dequantization (float16→float32, int8→float32)
    ↓ (aplicar scale/zero_point según metadata)
[4] SH Expansion (padding con ceros)
    ↓ (coeficientes truncados se rellenan)
[5] Pruning Reconstruction (optional index_map)
    ↓ (reconstruir gaussianas podadas si es necesario)
Datos descomprimidos (GaussianData + DeformationData)
```

**Detalles de cada substep**:

#### 2.1 Entropy Decoding (Lossless)
```python
# Pseudocódigo
compressed_bytes = read(archive)
for strategy in strategies_reversed:
    if strategy.name == "entropy_coding":
        decompressed_bytes = algorithm.decompress(compressed_bytes)
        # zlib.decompress(), zstd.decompress(), etc.
```

**Naturaleza**: **Completamente reversible (lossless)**. Los bytes descomprimidos son idénticos a los datos pre-compresión entrópica. No hay pérdida de información.

#### 2.2 HexPlane Decompression (Lossy/Lossless según método)
- **SVD (lossy)**: reconstruye W ≈ U × Σ_k × V^T (aproximación de rango k, error ||W - W_reconstructed||_F)
- **Quantize (lossy)**: convierte float16 → float32 (pérdida de precisión ~0.0005)
- Se preservan los gradients para posible fine-tuning (aunque en este caso es inferencia)

**Naturaleza**: Lossy si se usó SVD (pérdida controlada por rank k) o quantize (pérdida de precisión). La reconstrucción es aproximada.

#### 2.3 Dequantización (Lossy)
```python
# Para int8: recuperar valores aproximados (no exactos)
decompressed = (quantized - zero_point) × scale

# Para float16: simple casting (pérdida de precisión)
decompressed = float32(float16_value)
```

**Naturaleza**: **Lossy (irreversible)**. La cuantización es un mapeo con pérdida:
- float32→float16: Pérdida de 13 bits de mantisa (error relativo ≈ 0.0005)
- float32→int8: Cuantización afín introduce error de redondeo ≈ Δ/2 por bin
- Los valores descomprimidos son **aproximaciones**, no réplicas exactas

**Nota crítica**: Los factores de escala (scale) y punto cero (zero_point) se almacenan en el manifest y se aplican **por atributo**:
- xyz: float16 → float32
- opacity: float16 → float32
- scaling: float16 → float32
- rotation: float16 → float32 (quaternion)
- features_dc: float16 → float32
- features_rest: float16 → float32

#### 2.4 SH Expansion (Lossy)
Si se redujo de grado 3 a grado 1, al descomprimir:
```python
# Reconstrucción de coeficientes faltantes
features_rest_original = (N, 15, 3)  # grado 3
features_rest_compressed = (N, 3, 3)  # grado 1

# Restauración
features_rest_restored = np.zeros((N, 15, 3))
features_rest_restored[:, :3, :] = features_rest_compressed
# Los restantes (3:15) permanecen en 0.0
```

**Naturaleza**: **Lossy (irreversible)**. Los coeficientes SH de alto orden eliminados **no se pueden recuperar**. Se rellenan con ceros por defecto, lo que asume radiancia constante en direcciones de alto orden.

Esto introduce un **error controlado** predecible: los coeficientes de alto orden ausentes se asumen como cero, degradando la fidelidad de iluminación difusa pero manteniendo la geometría.

#### 2.5 Pruning Reconstruction (Lossy - Pérdida Permanente)
Solo aplicable si se usó PruningStrategy:
- Se carga el `index_map` desde el manifest
- `index_map[i] = j` indica que la gaussiana original i mapeó a j tras poda
- `index_map[i] = -1` indica gaussiana removida (no se recupera)
- La estructura de gaussianas podadas es **irreversible** (no se pueden recuperar)

**Naturaleza**: **Lossy (pérdida permanente)**. Las gaussianas eliminadas durante la poda están **perdidas para siempre**. No hay forma de recuperarlas durante la descompresión. El modelo descomprimido tendrá menos gaussianas que el original.

**Latencia típica de Fase 2**: 500 ms - 2 s (según algoritmo entropy coding y tamaño del modelo)

### Fase 3: Exportación per-frame con Deformación

**Objetivo**: Ejecutar la red de deformación para cada frame del video y generar secuencias PLY compatibles con SuperSplat.

**Proceso**:

```
Parámetros descomprimidos (GaussianData)
    ↓
Reconstrucción de red HexPlane
    |(load_state_dict con pesos descomprimidos)
    ↓
Para cada frame t ∈ [0, num_frames):
    ├─ t_normalized = t / (num_frames - 1)
    ├─ time_tensor = [[t_normalized]] (expandido a N gaussianas)
    ├─ Evaluación de red:
    │  means3D_final, scales_final, rotations_final, opacity_final, shs_final = 
    │      deformation_network(means3D, scaling, rotation, opacity, shs, time_tensor)
    ├─ Conversión a numpy
    └─ Escritura de PLY: time_XXXXX.ply
```

#### 3.1 Reconstrucción de la Red de Deformación

La red HexPlane se reconstruye en GPU a partir del state_dict descomprimido:

```python
# Pseudocódigo de decompress.py
deform_net = deform_network(hyper_args)
deform_net.load_state_dict(deformation.state_dict)
deform_net = deform_net.to("cuda")
deform_net.eval()
```

**Parámetros críticos inferidos del state_dict**:
- **net_width**: Ancho de la red (features ocultas) - inferido de la forma de pesos
- **defor_depth**: Número de capas fully connected
- **plane_coef**: Coeficiente de expansión de planos
- **time_significance**: Importancia de la variable temporal

El manifest JSON embebido contiene estos values exactos, permitiendo reconstrucción exacta sin necesidad de archivos de configuración adicionales.

#### 3.2 Evaluación de Red para Cada Frame

```python
with torch.no_grad():
    for frame_idx in range(num_frames):
        t_normalized = frame_idx / max(num_frames - 1, 1)
        time_tensor = torch.full((N, 1), t_normalized, device="cuda")
        
        # Evaluación de red (forward pass)
        means3D_d, scales_d, rots_d, opacity_d, shs_d = deform_net(
            means3D, scaling, rotation, opacity, shs, time_tensor
        )
        
        # means3D_d: posiciones deformadas (N, 3)
        # scales_d: escalas deformadas (N, 3)
        # rots_d: rotaciones deformadas (N, 4) [cuaterniones]
        # opacity_d: opacidades deformadas (N, 1)
        # shs_d: características SH deformadas (N, total_coeffs, 3)
```

**Características de la evaluación**:
- **Batch processing**: Todas las N gaussianas se procesan simultáneamente
- **Computational cost**: O(N × net_params) por frame
- **Memoria**: Típicamente <8 GB para modelos de hasta 500k gaussianas
- **Velocidad**: 10-50 ms por frame en GPU (dependiendo de arquitectura)

#### 3.3 Escritura de PLYs por Frame

Para cada frame, se genera un archivo PLY estándar 3DGS:

```python
_write_ply(
    path=f"time_{frame_idx:05d}.ply",
    xyz=means3D_d.cpu().numpy(),           # (N, 3)
    features_dc=shs_d[:, :n_dc, :].cpu().numpy(),    # (N, 1, 3)
    features_rest=shs_d[:, n_dc:, :].cpu().numpy(),  # (N, K, 3)
    opacity=opacity_d.cpu().numpy(),                 # (N, 1)
    scaling=scales_d.cpu().numpy(),                  # (N, 3)
    rotation=rots_d.cpu().numpy(),                   # (N, 4)
)
```

**Estructura del PLY para frame $t$**:
```

property float x

property float nx              (siempre 0, no usado)

```

**Latencia típica de Fase 3**: 
- Para 300 frames de un modelo de 100k gaussianas: 5-15 segundos
- Promedio: 10-50 ms/frame en GPU

## 3.4.2 Qué Altera en los Gaussianos al Renderizar

El proceso de renderizado de 4DGS en `gaussian_renderer/__init__.py` y `render.py` realiza múltiples transformaciones en los parámetros gaussianos antes de la rasterización. Es crucial entender estas transformaciones para comprender cómo las perturbaciones en parámetros comprimidos afectan el resultado visual final.

### 2.1 Evaluación de Armónicos Esféricos (SH)

El primer paso en el renderizado es **evaluar los coeficientes SH en la dirección de visión específica de cada cámara**.

**Fundamento teórico**:
Los armónicos esféricos parametrizan la radiancia saliente de un punto en todas las direcciones:

$$L(\theta, \phi) = \sum_{l=0}^{d} \sum_{m=-l}^{l} c_{l,m} Y_l^m(\theta, \phi)$$

donde:
- $(c_{l,m})$ son los coeficientes SH (almacenados en features_dc y features_rest)
- $Y_l^m$ son las funciones base esféricas harmónicas
- $(\theta, \phi)$ son las coordenadas esféricas de la dirección de visión

**Evaluación en GPU**:
```python
# En gaussian_renderer/__init__.py
shs_view = pc.get_features.transpose(1, 2).view(-1, 3, (pc.max_sh_degree+1)**2)
# shs_view: (N, 3, (D+1)²) donde D es el grado SH

dir_pp = pc.get_xyz - viewpoint_camera.camera_center
dir_pp_normalized = dir_pp / ||dir_pp||_2

sh2rgb = eval_sh(pc.active_sh_degree, shs_view, dir_pp_normalized)
colors_precomp = clamp_min(sh2rgb + 0.5, 0.0)
```

**Transformaciones aplicadas**:
1. **Normalización de dirección**: La dirección de visión se normaliza para evaluación esférica
2. **Evaluación SH**: $\text{eval\_sh}(\text{degree}, c, \vec{dir})$ calcula $\sum_l \sum_m c_{l,m} Y_l^m(\vec{dir})$
3. **Offset y clipping**:
   ```python
   colors_precomp = max(sh2rgb + 0.5, 0.0)  # offset para estabilidad numérica y clipping
   ```

**Impacto de la compresión**:
- Si se redujo SH degree (3→1): Coeficientes de alto orden ausentes → colores menos sensibles a la dirección de visión
- Si se cuantizó a float16: Error de ~0.0005 por coeficiente → cambios de color <1 nivel de 8-bit
- El offset de 0.5 proporciona **margen de tolerancia** a pequeños errores de cuantización

### 2.2 Transformaciones de Deformación Temporal

Durante el renderizado en modo "fine", la red de deformación altera **todos** los parámetros gaussianos:

```python
# En gaussian_renderer/__init__.py, render()
if "fine" in stage:
    means3D_final, scales_final, rotations_final, opacity_final, shs_final = pc._deformation(
        means3D, scales, rotations, opacity, shs, time
    )
```

**Parámetros transformados**:

#### 2.2.1 Posición (xyz)
- **Entrada**: $\vec{m} \in \mathbb{R}^{N \times 3}$ (posiciones en coordenadas del objeto)
- **Transformación**: $\vec{m}' = \vec{m} + \Delta \vec{m}(t)$, donde $\Delta \vec{m}$ es predicho por la red HexPlane
- **Rango típico**: $|\Delta \vec{m}| \lesssim 5\%$ de la escala de la escena
- **Impacto visual**: Movimiento de gaussianas, deformación geométrica

#### 2.2.2 Escala (scaling)
- **Entrada**: $\vec{s} = [\log(\sigma_x), \log(\sigma_y), \log(\sigma_z)] \in \mathbb{R}^{N \times 3}$ (log-escala)
- **Transformación**: $\vec{s}' = \vec{s} + \Delta \vec{s}(t)$
- **Activación después de la red**: $\vec{\sigma} = \exp(\vec{s}')$ (exponencial para garantizar positividad)

#### 2.2.3 Rotación (quaternion)
- **Entrada**: $\vec{q} \in \mathbb{R}^{N \times 4}$ (cuaterniones no normalizados)
- **Transformación**: $\vec{q}' = \vec{q} + \Delta \vec{q}(t)$
- **Activación después de la red**: $\vec{q}'' = \frac{\vec{q}'}{||\vec{q}'||_2}$ (normalización a cuaternión unitario)

#### 2.2.4 Opacidad (opacity)
- **Entrada**: $\vec{\alpha}_{\text{raw}} \in \mathbb{R}^{N \times 1}$ (parámetro raw)
- **Transformación**: $\vec{\alpha}_{\text{raw}}' = \vec{\alpha}_{\text{raw}} + \Delta \vec{\alpha}(t)$
- **Activación después de la red**: $\vec{\alpha} = \sigma(\vec{\alpha}_{\text{raw}'})$ (sigmoid para constrair a [0,1])

#### 2.2.5 Características SH
- **Entrada**: $\vec{c} \in \mathbb{R}^{N \times K \times 3}$ donde K = $(D+1)^2$ (D es grado SH)
- **Transformación**: $\vec{c}' = \vec{c} + \Delta \vec{c}(t)$
- **Sin activación**: Los coeficientes SH sin procesar pueden ser negativos (su interpretación es matemática, no física)

### 2.3 Activaciones de Parámetros

Después de la deformación, se aplican **activaciones no lineales** a ciertos parámetros:

```python
scales_final = pc.scaling_activation(scales_final)      # exp()
rotations_final = pc.rotation_activation(rotations_final)  # normalize
opacity = pc.opacity_activation(opacity_final)          # sigmoid
```

**Impacto en cascada**:
1. La red de deformación aprende a predecir parámetros "raw" sin restricciones
2. Las activaciones constringen estos valores al espacio válido
3. Pequeños cambios en los parámetros raw pueden tener efectos grandes post-activación

Ejemplo: Un cambio en log-scaling de ±0.1 genera:
```
exp(s ± 0.1) / exp(s) ≈ e^(±0.1) ≈ 1 ± 0.105  (±10.5% de cambio en escala)
```

### 2.4 Configuración de Rasterización

La rasterización de Gaussianas (mediante `diff_gaussian_rasterization`) requiere varios parámetros que se alteran según el frame:

```python
raster_settings = GaussianRasterizationSettings(
    image_height=H,
    image_width=W,
    tanfovx=tan(FoVx/2),
    tanfovy=tan(FoVy/2),
    bg=bg_color,
    scale_modifier=scaling_modifier,
    viewmatrix=world_view_transform,
    projmatrix=full_proj_transform,
    sh_degree=pc.active_sh_degree,           # Puede estar reducido
    campos=camera_center,
    prefiltered=False,
    debug=False
)
```

**Parámetro crítico**: `sh_degree = pc.active_sh_degree`
- Si se comprimió con SH reduction a grado 1, entonces `active_sh_degree = 1`
- El rasterizador solo evalúa hasta ese grado, ignorando coeficientes superiores (que serían 0 anyway)

### 2.5 Proyección 3D → 2D

Un paso crítico es la **proyección de posiciones 3D a espacio de pantalla (2D)**:

```
Posición 3D: (x, y, z) en coordenadas del mundo
    ↓
world_view_transform: (x, y, z) → (x', y', z') en coordenadas de cámara
    ↓
full_proj_transform: (x', y', z') → (u, v, d) en espacio de pantalla
    ↓
Posición 2D: (u, v) en píxeles [0, W) × [0, H)
```

Las deformaciones en posición 3D se propagan **no linealmente** a través de esta proyección:
```
∂(u,v) / ∂(x,y,z) = ∇(proj_transform)
```

Cambios pequeños en posición 3D pueden no ser visibles si están fuera de pantalla (culling de frustum) o si se proyectan a la misma región de píxeles.

### 2.6 Cálculo de Covarianza 3D

Antes de rasterizar, se calcula la **matriz de covarianza 3D** de cada gaussiana:

```python
if pipe.compute_cov3D_python:
    cov3D_precomp = pc.get_covariance(scaling_modifier)
else:
    scales = pc._scaling
    rotations = pc._rotation
    # El rasterizador calcula cov3D = R × S × S^T × R^T
```

Donde:
- R es la matriz de rotación construida a partir del quaternión
- S es la matriz diagonal de escalas
- Los valores post-activación aseguran que cov3D es **semidefinida positiva**

### Resumen de Transformaciones

Las alteraciones en los parámetros gaussianos durante renderizado forman una **cadena de transformaciones no lineales**:

```
Parámetros comprimidos
    ↓
Dequantización → float32
    ↓
Red de deformación (HexPlane)
    ↓
Activaciones (exp, sigmoid, normalize)
    ↓
Evaluación de SH
    ↓
Proyección 3D→2D
    ↓
Cálculo de covarianza
    ↓
Rasterización (splat rendering)
    ↓
Imagen final
```

**Cada paso introduce o amplifica errores potenciales**. Sin embargo, la **no linealidad de las funciones de activación proporciona cierta robustez**: pequeños errores se comprimen en regiones menos perceptuales.

## 3.4.3 Cómo Se Carga en SuperSplat

SuperSplat es un visor web de tiempo real para Gaussian Splatting que permite visualizar, editar y renderizar modelos en el navegador. La integración de modelos 4DGS descomprimidos requiere entender cómo SuperSplat interpreta el formato PLY y carga secuencias de frames.

### 3.1 Formato PLY Esperado

SuperSplat espera archivos PLY conformes al estándar 3DGS con la siguiente estructura de properties:

**Geometric Properties**:
```
x, y, z           : float (posición 3D)
nx, ny, nz        : float (normales, generalmente 0 - no usado en renderizado)
```

**Color Properties (Spherical Harmonics)**:
```
f_dc_0, f_dc_1, f_dc_2      : float (coeficientes DC)
f_rest_0...f_rest_44        : float (coeficientes de alto orden, grado 3)
                              o f_rest_0...f_rest_8 (grado 2)
                              o f_rest_0...f_rest_2 (grado 1)
                              o nada (grado 0)
```

**Appearance Properties**:
```
opacity           : float (opacidad raw, antes de sigmoid)
scale_0, scale_1, scale_2    : float (log-scaling)
rot_0, rot_1, rot_2, rot_3   : float (cuaternión, cualquier normalización)
```

**Full PLY Header Example** (para modelo comprimido con SH grado 1):

### 3.3 Carga de Secuencias Temporales

Para modelos 4DGS, SuperSplat interpreta múltiples archivos PLY como una **secuencia de frames**:

**Mecanismo de secuencia** (implementado en `supersplat/src/ply-sequence.ts`):
```typescript
// Pseudo-código TypeScript
async function loadPlyFrame(frameIndex: number): Promise<Splat> {
    const filename = `time_${String(frameIndex).padStart(5, '0')}.ply`;
    
    // Invoca el evento de importación
    const newSplat = await events.invoke('import', [{
        filename: filename,
        contents: file_data
    }]);
    
    // Destruye el frame anterior si existe
    if (sequenceSplat) {
        sequenceSplat.destroy();
    }
    
    sequenceSplat = newSplat[0];
    sequenceFrame = frameIndex;
    sequenceLoading = false;
    
    return newSplat[0];
}

// Control de animación
function playAnimation() {
    for (let t = 0; t < numFrames; t++) {
        frameTimeoutId = setTimeout(() => {
            loadPlyFrame(t);
            updateUI();
        }, (1000 / fps));  // Ejemplo: 30 fps
    }
}
```

**Flujo de carga**:
1. Usuario carga el primer frame (time_00000.ply)
2. SuperSplat detecta el patrón de numeración
3. **Almacena en caché** todos los archivos PLY de la secuencia
4. Al presionar Play, intercambia el modelo de GPU en cada frame
5. Renderizado en tiempo real sin reencargamiento de disco

### 3.4 Procesamiento de PLY en SuperSplat

Una vez cargado, SuperSplat convierte el formato PLY a su representación interna `Splat`:

```typescript
class Splat {
    splatData: SplatData {
        numSplats: number,
        properties: {
            position: Float32Array,      // xyz
            opacity: Float32Array,       // sigmoid(opacity_raw)
            scale: Float32Array,         // exp(log-scale)
            rotation: Quat[],            // normalize(quat)
            sh: Float32Array             // coeficientes SH reorganizados
        }
    }
}
```

**Transformaciones aplicadas durante import**:
1. **Lectura de PLY**: Parse del formato binario/ASCII
2. **Reorganización de memoria**: Arrays de structures → structures of arrays (SoA) para eficiencia en GPU
3. **Normalizaciones**:
   ```typescript
   // Opacidad: sigmoid(raw)
   opacity[i] = 1.0 / (1.0 + Math.exp(-opacityRaw[i]));
   
   // Escala: exp(log-scale)
   scale[i] = Math.exp(logScale[i]);
   
   // Rotación: normalizar cuaternión
   quat[i] = normalize(quatRaw[i]);
   ```

### 3.5 Renderizado en SuperSplat

Una vez cargadas las propiedades gaussianas, SuperSplat renderiza mediante:

```typescript
class SplatRenderer {
    render(camera: Camera, splat: Splat): WebGL2RenderingContext {
        // 1. Setup viewport según parámetros de cámara
        gl.viewport(0, 0, canvas.width, canvas.height);
        
        // 2. Clear framebuffer
        gl.clear(gl.COLOR_BUFFER_BIT | gl.DEPTH_BUFFER_BIT);
        
        // 3. Para cada gaussiana:
        for (let i = 0; i < splat.splatData.numSplats; i++) {
            // a. Obtener posición, escala, rotación, opacidad
            const pos = splat.properties.position[i];
            const scale = splat.properties.scale[i];
            const rot = splat.properties.rotation[i];
            const opacity = splat.properties.opacity[i];
            
            // b. Proyectar a 2D (usando matrices view/projection de cámara)
            const pos2D = project(pos, camera);
            
            // c. Calcular covarianza 2D (resultado de transformación de covarianza 3D)
            const cov2D = calculateCovariance2D(scale, rot, camera);
            
            // d. Renderizar quad con shader de gaussiana
            renderQuad(pos2D, cov2D, opacity, color);
        }
        
        // 4. Presentar resultado
        gl.swap();
    }
}
```

**Pipeline de shaders en SuperSplat**:

**Vertex Shader**:
```glsl
uniform mat4 viewMatrix;
uniform mat4 projMatrix;

attribute vec3 position;
attribute vec4 rotation;
attribute vec3 scale;
attribute float opacity;
attribute vec3 sh_dc;
attribute vec3 sh_rest[15];  // Grado 3

void main() {
    // Aplicar transformaciones de cámara
    vec4 pos_view = viewMatrix * vec4(position, 1.0);
    gl_Position = projMatrix * pos_view;
    
    // Pasar parámetros gaussianos al fragment shader
    vOpacity = opacity;
    vShDc = sh_dc;
    vShRest = sh_rest;
    vScale = scale;
    // ...
}
```

**Fragment Shader**:
```glsl
varying float vOpacity;
varying vec3 vShDc;
varying mat3 vCov2D;  // Covarianza de pantalla 2D

void main() {
    // Evaluar gaussiana 2D en coordenadas de píxel
    float gaussian_value = exp(-0.5 * (uv^T * inv(vCov2D) * uv));
    
    // Evaluar color desde coeficientes SH
    vec3 color = evaluateSH(vShDc, vShRest, viewDirection);
    
    // Aplicar opacidad
    gl_FragColor = vec4(color, vOpacity * gaussian_value);
}
```

### 3.7 Optimizaciones de Memoria SuperSplat

Para evitar sobrecarga de memoria, SuperSplat implementa:

#### 3.7.1 Lazy Loading
```
Load frame N
    ↓
Display
    ↓
En background, precarga frame N+1
    ↓
Usuario puede cambiar de frame instantáneamente
```

#### 3.7.2 Memory Pooling
```
Prealoca memory buffers para Splat
    ↓
En cada frame:
    Borra contenido de buffers
    Carga nuevo frame
    Reutiliza memoria (evita GC)
```

#### 3.7.3 GPU Streaming
```
Mantiene 2-3 frames en VRAM
Transfiere desde CPU cuando es necesario
Threshold de memoria configurable
```
